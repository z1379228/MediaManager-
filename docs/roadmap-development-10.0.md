# MediaManager 開發版 10.0 規劃

## 10.0：低耗用自檢 MOD 與重複流程整理

- 新增手動、唯讀的 Self Check MOD，檢查 MOD registry、啟用控制器、按鍵狀態、
  分頁顯示、依賴與安全模式是否一致。
- 自檢不下載媒體、不執行完整轉檔、不連線外部網站，也不在背景重跑 Pytest。
- 檢查結果分為通過、警告、阻擋，提供修復入口與可匯出的去識別報告。
- 依 MOD 狀態、設定與依賴指紋快取結果；只有相關事件變更才讓快取失效。
- 將重複的 FFmpeg、Deno／EJS、whisper-cli 與模型檢測整併到同一診斷服務。
- 評估把批次命名／輸出預設、下載後處理與開發者發布附件檢查拆成可選 MOD；
  安全模式、簽章、信任庫、路徑、佇列與統一啟用控制器保留在核心。

完成條件：一般自檢可在秒級完成且不啟動外部 provider；結果與實際 UI 狀態一致，
正式發布的完整 Ruff／Pytest／copied-folder／簽章檢查仍由 CI 與發布工具負責。

## 後續區塊式開發規則

- 每個開發大版本只設定一個核心主題，例如下載、搜尋或 MOD 適配；不在同一大版本同時
  大改三條主線。
- 主題內再依網站切成可獨立驗收的區塊，預設順序為 YouTube、Bilibili、通用網站；
  MEGA 等特殊協定維持獨立 MOD 與獨立區塊。
- 每個網站區塊必須一次完成 provider、UI、錯誤提示、啟用／停用流程與針對性測試，
  通過後才進入下一個網站。
- 下載佇列、格式模型、搜尋結果合約與歷史資料等跨網站能力留在核心；登入、網址規則、
  彈幕及網站特有解析留在對應 MOD，避免核心持續膨脹。
- 區塊未完成不得標記大版本完成，也不得把半成品帶入正式版候選。

建議後續主題順序為：先完成 10.0 自檢與開發門檻，再以一個大版本強化下載，下一個
大版本強化搜尋，之後才集中處理第三方 MOD 適配。各大版本內均依上述網站區塊逐一交付。
