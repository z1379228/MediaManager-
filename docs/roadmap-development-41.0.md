# Development 41.0 版本計畫

> 收斂狀態（2026-07-18）：`DONE BASELINE / SUPERSEDED`。本文件以下內容為歷史規劃；唯一 canonical 計畫為 `roadmap-development-34.0-40.0.md`。

歷史狀態（封存時）：規劃中。41.0 聚焦「測試資料隔離、稽核效率與歷史版本維護」，維持日常只稽核 current + previous 的規則。

## P0：稽核與資料隔離

- **測試資料隔離**：pytest、smoke、複製資料夾測試使用獨立暫存目錄，不把 UserData、Cookie 或本機工具寫入版本產物。
- **日常稽核**：current + previous 版本執行快速 SHA-256 稽核；大版本封版、資安事件或定期維護才啟用 full-history。
- **歷史保留**：GitHub Releases 的 EXE、SHA256SUMS.txt 與 release-info.json 不刪除，本機只保留目前版與上一版。
- **證據完整**：Ruff、回歸測試、網站／MOD／版本／文字污染稽核結果與阻擋項可追溯。

## P1：維護流程

- 暫存檔清理工具只處理專案允許範圍，保留 Version、.venv、.tool-cache、UserData 與使用者下載資料。
- 失敗測試顯示實際測試名稱、逾時與隔離目錄，避免把環境問題誤判成程式污染。
- 版本資訊、依賴清單、MOD 狀態與錯誤日誌分離保存，敏感欄位自動脫敏。
- 發布前後重新產生 SHA-256 與 release-info，驗證附件不被覆蓋或遺漏。

## P2：條件式候選

- 自動化稽核報告摘要、差異比較與保留版本清單先維持唯讀輸出。
- 新網站、Gopeed、P2P／Torrent 與通用解析器維持候選，不進入核心。
- 雲端備份與遠端稽核需另行審查隱私、憑證與使用者授權。

## 每輪純規劃驗收

1. 唯讀掃描工作樹、文件、參考清單、版本與 MOD 狀態。
2. 建立或補充下一個未完成版本計畫；若沒有新差異，記錄原因與下一輪候選。
3. 只記錄唯讀稽核結果，不修改程式碼、使用者資料或版本產物。
4. 正式簽章缺失、外部網站限制與歷史 UserData 項目維持為阻擋項。

41.0 不建立 Stable 或 Testing 產物；發布需另行取得使用者明確決定。
